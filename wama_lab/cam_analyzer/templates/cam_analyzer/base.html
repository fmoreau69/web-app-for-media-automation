WAMA Lab — Cam Analyzer

Outil de recherche pour l'analyse automatisée de vidéos embarquées sur la navette autonome Navya. Il combine le suivi d'objets YOLO, la télémétrie GPS RTMaps et l'analyse comportementale pour produire des rapports d'interactions navette/usagers de la route.

Pipeline d'analyse
1 Source

Vidéos caméras individuelles ou fichier RTMaps .rec (quadrature H264 800×500 + GPS NMEA)

2 Détection

YOLO tracking (BotSORT/ByteTrack) — véhicules, piétons, cyclistes.
SAM3 text-prompt (optionnel) — lignes de stop, passages piétons dans les fenêtres d'intersection.

3 Analyse

Fenêtres GPS, phases d'arrêt / insertion, ego-motion, D0 relatif, densité trafic.
SAM3 : flags stop_line_detected / crossing_detected dans les métadonnées de chaque segment.

4 Rapport

Segments temporels typés, export CSV/JSON, vidéos annotées

Types de rapports
Proximité & Dépassements

Détecte les suivis rapprochés (caméra arrière), les dépassements (caméras latérales) et les croisements de piétons (caméra avant). Ne nécessite pas de GPS.

Insertions aux intersections

Analyse la caméra avant dans les fenêtres GPS autour des intersections configurées. Détecte les véhicules à l'arrêt puis en insertion devant la navette. Nécessite GPS (RTMaps ou CSV API) + intersections GPS dans le profil.

Guide d'utilisation

Démarrage rapide
  1. Cliquez Nouvelle session et donnez-lui un nom.
  2. Sélectionnez ou créez un profil d'analyse (modèle YOLO, type de rapport, classes).
  3. Source vidéo : glissez des vidéos sur les zones caméra — ou utilisez Fichier RTMaps pour un .rec Navya.
  4. Cliquez Démarrer l'analyse et suivez la progression.
  5. Consultez les résultats dans l'onglet Résultats / Segments et exportez en CSV ou JSON.

  3. Optionnel : ajouter le CSV API (lat/lon décimal @1Hz, plus précis que le NMEA RTMaps).
  4. Cliquer Extraire — la progression s'affiche. Les vues avant/arrière sont créées automatiquement.
  5. Lancer l'analyse normalement.
Layout quadrature Navya : Arrière (haut-gauche), Avant (haut-droite), Gauche (bas-gauche), Droite (bas-droite) — 400×250 px par vue.
Configurer les intersections GPS
  1. Dans le profil, choisir le type Insertions aux intersections.
  2. Cliquer + Ajouter dans la section Intersections.
  3. Cliquer sur la mini-carte pour placer le point — lat/lon se remplissent automatiquement.
  4. Ajuster le rayon (m) : la navette déclenchera l'analyse quand elle sera dans ce rayon.
  5. Sauvegarder. Plusieurs intersections peuvent être configurées par profil.

    Architecture

    Application Django dans wama_lab/cam_analyzer/. Tâches lourdes exécutées en asynchrone via Celery + Redis. Résultats stockés en base PostgreSQL, vidéos annotées dans MEDIA_ROOT/cam_analyzer/.

        │   CameraView records créés

        └─[process_session_task]
            │
            ├─ Charge modèle YOLO (.pt)
            ├─ [Optionnel] Charge RoadSegmenter (modèle BDD100K)
            ├─ Pré-calcul _sam3_windows (si SAM3 activé + GPS + intersections)

            │

    Algorithme IntersectionAnalyzer

    Pour chaque intersection configurée, parcourt la piste GPS et délimite les périodes où la navette est dans le rayon radius_m via la formule de Haversine. Retourne une liste de fenêtres {t_enter, t_exit, gps_points}.

    Cherche MIN_STOPPED_FRAMES (3) frames consécutives où le déplacement du centre bbox est < seuil.
    Seuil = BASE_STOPPED_PX (5px) + shuttle_speed × SPEED_FACTOR (0.5 px/km·h⁻¹) — compensation ego-motion proportionnelle à la vitesse GPS de la navette.

    Après le premier mouvement détecté (déplacement > 2×BASE_STOPPED_PX) :
    Si masque routier disponible : vérifie si le pied du véhicule (bas-centre bbox) entre dans le polygone de chaussée sur ≥ MIN_INSERTION_FRAMES (2) des 10 frames post-mouvement.
    Sinon (heuristique centre) : vérifie si le véhicule se rapproche du centre de l'image et atteint la zone centrale (±35 % largeur).
    Résultats : insertion | wait | turn.

    • t0 : première détection du véhicule dans la fenêtre
    • t1 : début du mouvement post-arrêt
    • t2 : T2_STABLE_FRAMES (3) frames consécutives stables (< T2_STABLE_PX) dans la zone centrale → insertion complète
    • D0_relative = frame_height / hauteur_bbox médiane (proxy inverse de la distance)
    RoadSegmenter

    Module optionnel (utils/road_segmenter.py) activé si AnalysisProfile.road_model_path est renseigné et que le rapport est intersection_insertion. Il charge un modèle YOLO segmentation (ex. BDD100K) et produit des polygones de chaussée par frame. Ces polygones sont stockés dans DetectionFrame.detections avec type="road_mask" et consommés par IntersectionAnalyzer pour la détection d'insertion basée sur la géométrie réelle plutôt que l'heuristique de position horizontale.

    Point-in-polygon : ray casting pur Python — pied de bbox (bas-centre) vs. polygone chaussée. Compatible avec tout modèle dont les noms de classes contiennent : road, drivable area, direct drivable, alternative drivable, lane.

    SAM3RoadAnalyzer Phase Avancée

    Module optionnel (utils/sam3_road_analyzer.py) activé si AnalysisProfile.sam3_markings_enabled est coché et que le rapport est intersection_insertion. Il charge le modèle SAM3 image (même modèle que l'anonymiseur) via Sam3Processor et n'effectue l'inférence que sur les frames tombant dans les fenêtres GPS d'intersection pré-calculées avant la boucle caméra — réduisant à ~5–10 % des frames le coût SAM3 (10–50× plus lent que YOLO).

    Détection marquages

    Pour chaque prompt {label, prompt}, interroge SAM3 sur la frame. Les masques renvoyés (score ≥ 0.30) sont convertis en bbox et stockés dans DetectionFrame.detections avec type="sam3_marking".

    Fallback routier

    Si sam3_as_road_fallback = True et aucun modèle BDD100K n'est chargé, SAM3 génère un masque de chaussée (drivable road surface area in front of the vehicle) par frame, compatible avec _classify_post_stop_v2().

    YOLO (~2 GB VRAM) + SAM3 (~12–16 GB VRAM) — compatible RTX 4090 24 GB. SAM3 est déchargé après chaque caméra via sam3_analyzer.unload().
